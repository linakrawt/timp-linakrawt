# 1 Введение

В рамках курсовой работы разрабатывается онлайн-платформа и клиентское приложение к нему «Личный угол». Система реализует клиент-серверную архитектуру, где клиентом для пользователя является мини-версия браузера на основе Chromium, а сервером является сама онлайн-платформа (сайт), что позволяет использовать в качестве клиента другой браузер, который соответствует минимальным техническим требованиям.

Онлайн-платформа «Личный угол» — это сайт, использующий различные модули (приложения) для того, чтобы предоставить пользователю возможность использования функционала других сервисов, таких как Яндекс.Диск или авторизация через социальные сети, с помощью интеграции, а также использующий разработанные приложения для увеличения возможностей пользователей.

Полезность данного проекта заключается в минимизации используемой оперативной памяти при почти полноценном использовании сервисов, которые не связаны друг с другом. Также компактность и минималистичность дизайна позволяют меньше уставать пользователю при использовании данной платформы. Основной идеей работы является объединения различных сервисов в одном месте для удобства пользователя.

# 2 Архитектура системы

В архитектуре системы есть клиентская и серверная части.

Сервером является предоставляемый фреймворком DjangoHHTP-сервер для тестирования. Функционал сервера написан на языке программирования Python. Сервер отвечает за рендеринг (визуализацию) шаблонов, обработку запросов., поступающих от клиента, и реализацию функционала, разработанного для данного проекта.

Шаблоны являются графическим представлением для клиента, они написаны на языке разметки HTML и используют стили, написанные на языке таблиц стилей CSS. CSS используется для улучшения графического представления (дизайна).

Также были использованы коды на языке программирования JavaScript для увеличения количества получаемой сервером информации и увеличения возможностей клиента для пользователя.

Клиентом может являться любая программа способная отправлять запросы серверу по URL адресам и обрабатывать, полученные в ответ данные. В данной работе клиентом могут являться браузеры, которые соответствуют требованиям, описанным в техническом задании, а также есть созданное сторонними сервисами приложение мини-браузер для ОС Windows на основе Chromium, которое отображает страницу по адресу localhost (127.0.0.1) с портом 8000. Приложение можно настроить, изменив начальный адрес и порт, для работы в локальной сети с других десктопных устройств. Описание настройки представлено в соответствующем разделе. Сам клиент отвечает за отображение пользовательского интерфейса и взаимодействия сервера и пользователя.

В качестве базы данных используется SQLite, автогенерируемая на стороне сервера и заполняемая им же с помощью определенных команд.

# 3 Описание протокола взаимодействия клиента и сервера, описание функциональности, предоставляемых сервером

Сервер обрабатывает запросы от клиента и предоставляет различные виды данных в зависимости от типа запроса. В данном проекте основными запросами, обрабатываемыми сервером, являются POST и GET. При обычных условиях GET запрос отвечает за получение ресурса, а POST запрос – за создание ресурса, но в целях увеличения удобства разработки было принято решение, что запрос POST будет означать обработку ресурса всеми способами, которые также включают PUT (обновление ресурса), DELETE (удаление ресурса) и другие виды запросов.

Протоколом взаимодействия клиента и сервера является HTTP, который построен на базе сообщений, называемых запрос (Request) и ответ (Response). Каждое сообщение состоит из трех частей: стартовая строка, заголовки и тело. Стартовая строка является единственной обязательной частью сообщения. Запросы, используемые в проекте, являются методами данного протокола.

Функциональность, предоставляемая сервером, заключается в реализация функционала, предоставляемого сторонними библиотеками и приложениями, где под приложениями подразумеваются подключаемые к фреймворку Django модули, выполняющие те или иные функции.

Сервер занимается обработкой HTML шаблонов и отправкой их на клиент в сообщении Response. Обрабатывает поступающие данные в сообщении Request с помощью представлений (Views), форм (Forms) и моделей (Models).

Сервер предоставляет пользователю следующие возможности:

1. Авторизация и регистрации с помощью встроенной в Django системы создания и работы с новыми пользователями.
2. Выход из текущей учетной записи для входа в другую.
3. Возможность регистрироваться и авторизовываться с помощью социальных учётных записей пользователя по средству использования стороннего модуля (приложения Django) под названием Django-allauth. На данный доступны к подключению аккаунты с сервисов Одноклассники, ВКонтакте, Google, Яндекс.
4. Возможность подключения к уже существующему аккаунту социальных учетных записей при условии, если они не подключены к другим учетным записям на сервере.
5. Возможность отсоединения социальных учетных записей при условии, что у пользователя есть пароль.
6. Изменение пароля пользователя.
7. Установка пароля для аккаунта пользователя, если он отсутствует. При регистрации с помощью социальных учетных записей пароль отсутствует по умолчанию.
8. При регистрации пользователя обязателен E-mail, поэтому при создании нового социального аккаунта сервер позволяет автозаполнять данное поле и поле имени пользователя, если сервис выбранной социальной учетной записи возвращает подходящие данные.
9. При регистрации введённый E-mail требуется подтвердить (данную опция можно изменить в настройках сервера), для этого Django использует STMP-сервер на определенном почтовом сервисе. В данной работе используется учетная запись, созданная для курсовой работы, на сервисе onionmail.
10. К аккаунту пользователя можно добавлять другие адреса электронной почты, если они не зафиксированы за другими аккаунтами.
11. Пользователю может пользоваться приложениями «Дневник», «Календарь» и «Заметки». Сервер позволяет обновлять дневник, автозаполнять дневник сохраненными данными, создавать заметки, удалять заметки, просматривать даты по календарю.
12. Сервер возвращает имя текущего пользователя для отображения его клиентом.
13. Если пользователь имеет аккаунт суперпользователя, то ему разрешается доступ к разделам загрузки файла на сервер и администрирования сервера (предоставляется возможностями Django).
14. При наличии у пользователя установленного браузера и соответствующего ему драйвера пользователю предоставляется возможность создания скриншота сайта по его адресу в сети.
15. Пользователь может использовать Яндекс.Карты в ограниченном функционале.
16. Пользователю предоставляется доступ к разделу Яндекс.Диск, если у него присоединена учетная запись Яндекс. В данном разделе сервер предоставляет возможность закачивать файлы на Яндекс.Диск, просматривать содержимое Яндекс.Диска, создавать директории на нем и удалять ресурсы (папки и файлы). Также предоставляется функция очистки корзины Яндекс.диска. Данный функционал реализуется с помощью сторонней библиотеки YaDisk для языка Python.
17. При наличии у аккаунта социальных учетных записей сервер предоставляет в разделе Профиль данные разного количества, получаемые от сервисов учетных записей.
18. Получение ответов от сервера в виде сообщений, возникающих при загрузке страницы.
19. Базовая реализация интернационализации сайта по средству использования команд для создания файлов, содержащих переводы под указанные в настройках локали. В данной работе была настроена реализация перевода сайта с английского языка на русский.

# 4 Краткое описание используемых инструментов

Для разработки были использованы в качестве инструментов языки программирования JavaScript и Python, язык разметки HTML, язык таблиц стилей CSS, база данных SQLite, фреймворк Django на основе языка Python и предоставляемый фреймворком сервер.

JavaScript - мультипарадигменный язык программирования. Поддерживает объектно-ориентированный, императивный и функциональный стили. В данном проекте JavaScript используется как встраиваемый язык для программного доступа к объектам приложений. Он применяется для придания интерактивности веб-страницам. Программы на этом языке называются скриптами. В данной работе они встроены в HTML и выполняются автоматически при загрузке веб-страницы. Скрипты распространяются и выполняются, как простой текст. Им не нужна специальная подготовка или компиляция для запуска.

HTML - стандартизированный язык разметки веб-страниц. Код HTML интерпретируется браузерами, и полученная в результате интерпретации страница отображается на экране монитора компьютера или мобильного устройства.

CSS - формальный язык описания внешнего вида документа (веб-страницы), написанного с использованием языка разметки HTML. Отделяя стиль представления документов от содержимого документов, CSS упрощает создание веб-страниц и обслуживание сайтов. Каскадные таблицы стилей описывают правила форматирования элементов с помощью свойств и допустимых значений этих свойств. Для каждого элемента использовались ограниченные наборы свойств, остальные свойства не будут оказывать на элемент никакого влияния.

Python - высокоуровневый язык программирования общего назначения, ориентированный на повышение производительности разработчика и читаемости кода. Синтаксис ядра Python минималистичен. В то же время стандартная библиотека включает большой набор полезных функций. Язык имеет высокую адаптивность, кроссплатформенность.

SQLite - компактная встраиваемая реляционная база данных. Встраиваемость означает, что большинство СУБД являются самостоятельными приложениями, взаимодействие с которыми организовано по принципу клиент-сервер. Программа-клиент посылает запрос на языке SQL, СУБД, которая в том числе может находиться на удаленном компьютере, возвращает результат запроса. SQLite является написанной на языке C библиотекой, которую динамически или статически подключают к программе.

Django - свободный фреймворк для веб-приложений на языке Python, использующий шаблон проектирования MVC. Сайт на Django строится из одного или нескольких приложений, которые рекомендуется делать отчуждаемыми и подключаемыми. Это одно из существенных архитектурных отличий этого фреймворка от других. Один из основных принципов фреймворка — Don&#39;t repeat yourself. Также, в отличие от других фреймворков, обработчики URL в Django конфигурируются явно при помощи регулярных выражений. Для работы с базой данных Django использует собственный ORM, в котором модель данных описывается классами Python, и по ней генерируется схема базы данных.

Сервер разработки Django, также называемый «runserver», по имени команды, которая его запускает — это встроенный лёгкий веб сервер, который используется в процессе разработки сайта. Он включен в Django для того, чтобы можно было быстро приступить к разработке сайта без траты времени на конфигурирование боевого веб сервера (Apache) раньше времени. Сервер разработки отслеживает изменения в коде и автоматически перезагружает его, помогая видеть вносимые изменения без перезагрузки веб сервера.

# 5 Детали реализации системы

В ходе разработки системы были выделены 2 основных этапа: разработка серверной части и клиентской части.

Серверная часть включает в себя приложения Django, которые включают в себя представления, формы, модели и URL адреса, по которым может переходить клиент. Для реализации функционала были использованы сторонние приложение Django и библиотеки для языка Python.

Список использованных сторонних приложений и библиотек:

1. Приложение django-allauth – предоставляет функционал для создания социальных учетных записей, визуализации процессов работы с данными аккаунта пользователя, получения токенов доступа сервисов, которые предоставляют данные для регистрации.
2. Библиотека YaDisk для Python – предоставляет возможность пользования функционалом сервиса Яндекс.Диск без реализации визуализации для фреймворка Django.
3. Библиотека selenium для Python – это драйвер браузера, то есть не имеющая пользовательского интерфейса программная библиотека, которая позволяет другим различным программам взаимодействовать с браузером, управлять его поведением, получать от браузера какие-то данные и заставлять браузер выполнять какие-то команды. В данной работе она были использована для получения скриншотов сайта по URL адресу.
4. Приложение Django-naomi – это серверная часть электронной почты для Django, которая позволяет просматривать электронную почту из веб-браузера вместо того, чтобы отправлять ее через SMTP-сервер. В данной работе при настройке сервера можно определить какой бэкенд для отправления почты использовать: Django-naomi или через SMTP-сервер.

Для серверной части также были созданы 2 новых приложения: auth\_first и upload.

Представления приложений используются для отображения шаблонов, которые в последующем используются клиентом. Они реализуют функционал за счет подключения форм, которые посылаются клиенту вместе с шаблонами и обрабатываются представлениями, выполняя определенные действия, для возврата ответа, содержащего либо какие-то данные, либо перенаправление на другую страницу. В рамках проекта были реализованы представления для шаблонов (шаблоны можно посмотреть в репозитории GitLab в поддиректории папки TiMPtemplates). Шаблоны используются для графического отображения функционала, описанного в разделе 3. Для работы с СУБД SQLite используются модели в приложениях, они отвечают за создание новых таблиц в базе данных и предоставляют данные таблиц представлениям для последующей возможности добавления, обновления, выборки или удаления данных из базы данных. URL адреса используются для связывания действий клиента с определенными представлениями. Формы – это объекты классов, которые позволяют отображать поля ввода клиенту и отправлять полученную информацию представлениям.

Для реализации регистрации в социальных сетях были использованы стандартные представления приложения django-allauth. Они записывают данные в базу данных в зависимости от действий пользователя на стороне клиента. Полученные данные позже используются приложениями auth\_first и upload для произведения операций по реализации функционала, указанного в техническом задании.

Приложение auth\_first использует эти данные для представления собранной информации о профиле человека в шаблоне с помощью выборки из базы данных по id пользователя данных в формате JSON.

Приложение upload использует данные для реализации проверки доступа к разделу сайта Яндекс.Диск и для получения токенов доступа, предоставляемых сервисом Яндекс. Затем используется библиотека YaDisk, которая с помощью полученного токена подключается к конкретному Яндекс.Диску и предоставляет набор методов для взаимодействия с ним. Таким образом были реализованы представления для реализации функциональности, описанной в разделе 3.

Django-allauth также используется для настроек текущего пользователя. Это подразумевает возможность изменения пароля, E-mail, установки пароля. Этот функционал также описан в разделе 3.

Для работы django-allauth требуются данные API приложений каждого сервиса, который нужно подключить к регистрации. Под данными подразумевается клиентский id, ключ и секретный ключ. Эти данные предоставляются разработчику API на соответствующих сервисах. После получения данные нужно добавить в базы данных с помощью администрирования Django. Данная процедура будет описана в разделе по настройке сервера.

Дневник, календарь и заметки – это представления, которые реализованы внутри приложения upload. Они используют базовый функционал Django и скрипты JavaScript для взаимодействия с базой данных и пользователем. Скрипты JavaScript использует из перечисленных выше представлений только календарь для улучшения интерактивности между пользователем и клиентом.

Для реализации Яндекс.карт требовался ключ API приложения, который создается другим способ в отличие от API приложений для получения данных профиля и доступа к Яндекс.Диск. Всю последующую реализацию функционала делает JavaScript и другие подключаемые со стороны скрипты.

Создание скриншотов сайта требует наличия у хоста браузера, который был определен при работе с библиотекой selenium, и загруженного драйвера, соответствующей версии для выбранного браузера. Библиотека запускает браузер в скрытом режиме, делает скриншот, выключает браузер и возвращает скриншот в шаблон вместе с ответом.

Загрузка на сервер собирает полученные из форм данные и добавляет их на сервер с помощью функций модели.

Представления на стороне сервера также занимаются переадресацией пользователя в случае, если он перейдет, будучи неавторизованным, по адресу сайта, где располагается какой-либо функционал. Также происходит перенаправление с адресов сайта в случае, если для них не предусмотрены графические отображения и они просто занимаются обработкой данных в представлениях.

На этом оканчивается описания функционала серверной части.

Клиентская часть использует HTML шаблоны с подключаемыми скриптами JavaScript и стилями CSS. Большая часть скриптов и стилей располагается в базовом шаблоне (его также можно посмотреть в репозитории GitLab по пути TiMP/templates/account/base.html).

Одним из важных модулей, который используется для загрузки файлов является Dropzone, он подключается в базовом шаблоне с помощью стилей и скриптов. Данный модуль позволяет создавать настраиваемую Drug&amp;Drop зону. Это было нужно для облегчения процесса загрузки и улучшения обработки данных.

Клиентом являются как описывалось ранее браузер и приложение, эмулирующее работу браузера с урезанным функционалом. Основной принцип работы заключается в интерактивном взаимодействии с элементами страницы, предоставленной и отрендеренной сервером. Нажатия на те или иные кнопки или ссылки отправляют запрос GET, который обрабатывается соответствующим представлением в зависимости от URL адреса, который возвращает активированный элемент шаблона.

На этом заканчивается описания функционала клиентской части.

# 6 Описание системы тестирования

# 7 Инструкция по настройке сервера

Настройка сервера представляет собой манипулирование разными параметрами в файле setting.py, а также прямое изменение кода внутри представлений или же установленных библиотек.

После скачивания с репозитория GitLab сервера, нужно установить библиотеки, указанные в файле requirements.txt, в предварительно созданное виртуальное окружение языка Python. После этого сервер можно запускать командой python manage.py runserver 0.0.0.0:8000. Но данные действия не позволят полностью использовать функционал сервера, чтобы он работал, так как изначально был разработан нужно изменить некоторые настройки в проекте. Обязательным условием является изменение файла в библиотеке Django-allauth по следующему пути: имя\_вашего\_окружения/Lib/site-packages/allauth/socialaccount/providers/yandex/views.py. Изменить нужно лишь один параметр – это URL авторизации, так как разработчики приложения иностранцы и они используют домен .com. Поменять нужно на .ru. Рисунок 7.1 показывает, как должен выглядеть конечный результат. Далее требуется создать и заполнить базу данных таблицами. Для этого используется команда python manage.py migrate. Далее стоит создать суперпользователя, так как иначе давать права суперпользователя без использования администрирования Django будет возможно лишь с помощью изменения базы данных. Для этого используется команда python manage.py createsuperuser. После ввода команды нужно будет заполнить поля формы регистрации в консоли и суперпользователь будет создан. В случае если при запуске сервера на клиенте все будет на английском языке, стоит использовать команды python manage.py makemessages и python manage.py compilemessages. Сначала применить первую команду затем поменять содержимое файла django.po на то, что находится внутри такого же файла в репозитории. На рисунке 7.2 представлено расположение нужных файлов в проекте относительно папки TiMP. Затем применить 2 команду. Тогда сайт переведется на русский язык.

Далее идет подключение социальных приложений. Это нужно для работы приложения Django-allauth и прочих, которые его используют. Чтобы добавить clientid, key, secretkey для конкретного сервиса из тех, что описаны в разделе 3, нужно создать в администрировании записи содержащие социальные приложения. Чтобы перейти в администрирование при получении ошибок от сервера из-за отсутствия социальных приложении, нужно перейти по адресу [http://127.0.0.1:8000/admin/](http://127.0.0.1:8000/admin/) , указать свои данные суперпользователя и ввести данные социальных приложений, которые использовались для данной курсовой или любые другие, полученные от сервисов, если были созданы соответствующие API. На рисунках 7.3-7.8 представлен порядок установки данных для социальных приложений. Эти действия нужно выполнить в обязательном порядке.
